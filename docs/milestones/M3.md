# Milestone 3: Email and SMTP

## Scope

M3 adds SMTP relay configuration, an injectable mailer abstraction, optional email addresses on
user accounts with a verification flow, and password reset via email.

Jaunder does not perform direct mail delivery or MX lookups. It hands outbound messages to a
configured relay (e.g. Postfix, SendGrid, Mailgun's SMTP endpoint) which is responsible for final
delivery.

### SMTP configuration

SMTP settings are stored as site_config keys:

| Key | Type | Description |
|---|---|---|
| `smtp.host` | `TEXT` | Relay hostname |
| `smtp.port` | `INTEGER` | Port number |
| `smtp.tls_mode` | `TEXT` | `"plain"`, `"starttls"`, or `"tls"` |
| `smtp.username` | `TEXT` | Optional credentials username |
| `smtp.password` | `TEXT` | Optional credentials password |
| `smtp.sender` | `TEXT` | Sender address (e.g. `"Jaunder <noreply@example.com>"`) |

`SmtpTlsMode` enum (`Plain`, `StartTls`, `Tls`) with `FromStr`/`Display` lives in `common`.
`SmtpConfig { host, port, tls_mode, username, password, sender }` and
`load_smtp_config(store: &dyn SiteConfigStorage) -> Option<SmtpConfig>` (returns `None` when
`smtp.host` is absent) also live in `common`.

### `Email` trait

An `Email` trait in `common` abstracts over the content of an outbound message, decoupling message
construction from transport:

```rust
pub trait Email: Send + Sync {
    fn recipient(&self) -> &str;
    fn subject(&self) -> &str;
    fn body_text(&self) -> &str;
}
```

`SimpleEmail { to: String, subject: String, body_text: String }` is the baseline implementation.
When HTML email is introduced in a later milestone, new types implement `Email` without modifying
`send_email` or existing implementors.

### `MailSender` trait and implementations

`MailSender` (in `common`) sends an `Email`:

```rust
#[async_trait]
pub trait MailSender: Send + Sync {
    async fn send_email(&self, email: &dyn Email) -> Result<(), MailError>;
}
```

`MailError` (in `common`): `NotConfigured` (SMTP not set up), `Send(String)` (transport error).

Three implementations:

- **`NoopMailSender`** (in `common`) — returns `MailError::NotConfigured`. Used when SMTP config is
  absent at startup.
- **`LettreMailSender`** (in `server`) — builds a `lettre` transport from `SmtpConfig` at startup,
  converts the `Email` to a `lettre::Message`, and sends it.
- **`CapturingMailSender`** (in `common`, behind a `test-utils` feature) — stores sent messages in
  a `Mutex<Vec<CapturedMail>>` for assertions in integration tests across crates.

`AppState` gains `mailer: Arc<dyn MailSender>`. `open_database` selects `NoopMailSender` or
`LettreMailSender` based on whether `load_smtp_config` returns `Some`.

### `jaunder smtp test` command

`jaunder smtp test --to <email>` loads SMTP config from the database, returns an error if SMTP is
not configured, otherwise sends a test message to the given address and reports success or failure.

### Email on user accounts

The `users` table gains two columns:

```sql
email          TEXT,
email_verified INTEGER NOT NULL DEFAULT 0
```

`UserRecord` gains `email: Option<String>` and `email_verified: bool`. `UserStorage` gains
`set_email(user_id: i64, email: Option<&str>, verified: bool) -> sqlx::Result<()>`.

A new table stores pending verification tokens (hash stored, raw token delivered by email):

```sql
email_verifications(
    token_hash  TEXT PRIMARY KEY,
    user_id     INTEGER NOT NULL REFERENCES users(user_id),
    email       TEXT NOT NULL,
    created_at  TEXT NOT NULL,
    expires_at  TEXT NOT NULL,
    used_at     TEXT
)
```

Timestamps are RFC 3339 text in SQLite, `chrono::DateTime<Utc>` in Rust (consistent with M2).

`EmailVerificationStorage` trait (in `common`):
- `create_email_verification(user_id: i64, email: &str, expires_at: DateTime<Utc>) -> sqlx::Result<String>` — stores hash, returns raw token.
- `use_email_verification(raw_token: &str) -> Result<(i64, String), UseEmailVerificationError>` — validates, marks used, returns `(user_id, email)`.

`UseEmailVerificationError`: `NotFound`, `Expired`, `AlreadyUsed`.

A user can have at most one pending verification at a time. Creating a new verification supersedes
any prior unverified token (the old token becomes `Expired` or `NotFound` on next use; no explicit
deletion is required).

### Password reset

A new table stores password reset tokens:

```sql
password_resets(
    token_hash  TEXT PRIMARY KEY,
    user_id     INTEGER NOT NULL REFERENCES users(user_id),
    created_at  TEXT NOT NULL,
    expires_at  TEXT NOT NULL,
    used_at     TEXT
)
```

`PasswordResetStorage` trait (in `common`):
- `create_password_reset(user_id: i64, expires_at: DateTime<Utc>) -> sqlx::Result<String>`
- `use_password_reset(raw_token: &str) -> Result<i64, UsePasswordResetError>` — returns `user_id`.

`UsePasswordResetError`: `NotFound`, `Expired`, `AlreadyUsed`.

`UserStorage` gains `set_password(user_id: i64, new_password: &str) -> sqlx::Result<()>` (hashes
internally via `spawn_blocking`, consistent with `create_user`).

Reset flow:
1. User submits username on `/forgot-password`.
2. Server looks up user by username. If no verified email is on file (user not found, or found but
   no verified email), returns an error telling the user to contact the site operator. Otherwise
   creates a token, sends a link to the verified email, and returns success. The success response
   does not confirm whether the username exists (avoids enumeration).
3. The link goes to `/reset-password?token=...`. User enters a new password.
4. `confirm_password_reset` validates the token, sets the new password, revokes all existing
   sessions for that user, and redirects to `/login`.

### Web UI

Four new routes:

| Path | Component | Auth required |
|---|---|---|
| `/profile/email` | `EmailPage` | Yes |
| `/verify-email` | `VerifyEmailPage` | No |
| `/forgot-password` | `ForgotPasswordPage` | No |
| `/reset-password` | `ResetPasswordPage` | No |

`EmailPage`: shows current email and verification status. Form to submit a new email address, which
calls `request_email_verification` — creates a token and sends the verification link. On submit the
page confirms "check your email."

`VerifyEmailPage`: reads `token` from the query string, calls `use_email_verification`, then
`set_email(..., verified: true)`, and renders success or an appropriate error (expired, already
used, not found).

`ForgotPasswordPage`: username form. On submit calls `request_password_reset` (see reset flow
above).

`ResetPasswordPage`: reads `token` from the query string, shows a new-password form. On submit
calls `confirm_password_reset`.
