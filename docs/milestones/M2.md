# Milestone 2: User Management

## Scope

### AppState

M2 introduces `AppState` — a concrete struct bundling all injectable handles — replacing the bare
`Arc<dyn SiteConfigStorage>` that `open_database` currently returns. This scales cleanly as
milestones add new subsystems.

```
AppState {
    site_config: Arc<dyn SiteConfigStorage>,
    users:       Arc<dyn UserStorage>,
    sessions:    Arc<dyn SessionStorage>,
    invites:     Arc<dyn InviteStorage>,
}
```

All storage implementations share the same `SqlitePool`. `open_database` and
`open_existing_database` return `Arc<AppState>`. `create_router` accepts `Arc<AppState>`, injects
it as a Leptos context value via `provide_context`, and also registers it as an Axum `Extension`
so that Axum extractors can reach it.

### User accounts

Each user account has a synthetic integer primary key (`user_id`), a unique lowercase username, an
argon2-hashed password, an optional display name, an optional bio, a creation timestamp, and a
`last_authenticated_at` timestamp updated on every successful login.

```
users(
    user_id               INTEGER PRIMARY KEY,
    username              TEXT NOT NULL UNIQUE,
    password_hash         TEXT NOT NULL,
    display_name          TEXT,
    bio                   TEXT,
    created_at            TEXT NOT NULL,
    last_authenticated_at TEXT
)
```

Timestamps are stored as RFC 3339 text and represented in Rust as `chrono::DateTime<Utc>`.

Usernames must match `[a-z0-9_-]+` (the ActivityPub-compatible character set). Rather than a
validation predicate, a `Username` newtype is used: it implements `FromStr` and rejects invalid
strings at parse time, returning `InvalidUsername`. Interior code works only with `Username` values
and never with raw strings; the trait methods accept `&Username`.

`UserRecord { user_id: i64, username: Username, display_name: Option<String>, bio: Option<String>, created_at: DateTime<Utc>, last_authenticated_at: Option<DateTime<Utc>> }` is the
type returned by `UserStorage` queries. It does not expose `password_hash` — that field is only
accessed inside `SqliteUserStorage`.

`CreateUserError` covers `UsernameTaken` and `Internal`. (`InvalidUsername` is a standalone parse
error on `Username`, not a `CreateUserError` variant — `create_user` receives an already-validated
`&Username`.) `UserAuthError` covers `InvalidCredentials` (username not found or password wrong —
no distinction is exposed to the caller) and `Internal`.

Password hashing and verification use the `argon2` crate (Argon2id, default parameters) and are
encapsulated entirely within `SqliteUserStorage`. `create_user` accepts a plain password and hashes
it internally via `tokio::task::spawn_blocking`. `authenticate` looks up the stored hash, verifies
the supplied password via `spawn_blocking`, and on success updates `last_authenticated_at` within
the same transaction.

### Sessions and authentication

A session is created at login and identified by an opaque token: 32 random bytes encoded as
base64url without padding (43 characters). Only the SHA-256 hash of the token, base64url-encoded,
is stored; the raw token is returned to the caller once and never persisted. Token generation is
factored into a `generate_token() -> String` free function in `auth.rs`, shared with M3's invite
and password-reset token generation.

```
sessions(
    token_hash   TEXT PRIMARY KEY,
    user_id      INTEGER NOT NULL REFERENCES users(user_id),
    label        TEXT,
    created_at   TEXT NOT NULL,
    last_used_at TEXT NOT NULL
)
```

`SessionStorage` exposes `create_session(user_id, label) -> sqlx::Result<String>` (returns the raw
token), `authenticate(raw_token) -> Result<SessionRecord, SessionAuthError>` (hashes the input,
looks up the row, updates `last_used_at`, returns the record), `revoke_session(token_hash) ->
sqlx::Result<()>`, and `list_sessions(user_id) -> sqlx::Result<Vec<SessionRecord>>`.
`SessionAuthError` covers `InvalidToken` and `SessionNotFound`. `SessionRecord { token_hash: String,
user_id: i64, username: Username, label: Option<String>, created_at: DateTime<Utc>, last_used_at: DateTime<Utc> }` is populated by joining `sessions` and
`users`.

Authentication is exposed as an Axum extractor `AuthUser { user_id: i64, username: Username, token_hash: String }`
implementing `FromRequestParts`. It reads the raw token from the `session` cookie or the
`Authorization: Bearer` header (cookie takes precedence), then calls
`state.sessions.authenticate`. Inside Leptos server functions, authentication is performed via
`leptos_axum::extract::<AuthUser>().await`. A `require_auth()` free function in `auth.rs` wraps
this and returns `ServerFnError` when no valid session is present.

The login server function sets a `session=<raw_token>; HttpOnly; SameSite=Lax; Path=/; Secure`
cookie via `leptos_axum::ResponseOptions` and also returns the raw token in the response body for
API clients. The register server function does the same: on success it creates a session, sets the
`session` cookie, and returns the raw token so API clients receive it without a secondary login
call. The logout server function revokes the session and clears the cookie by setting it to an
empty value with `Max-Age=0`.

### Invite codes

When the registration policy is `invite_only`, new accounts require a valid, unused, non-expired
invite code. Invite codes are single-use and time-limited; they are generated before the invited
user exists, so no user identity is associated at creation time.

```
invites(
    code       TEXT PRIMARY KEY,
    created_at TEXT NOT NULL,
    expires_at TEXT NOT NULL,
    used_at    TEXT,
    used_by    INTEGER REFERENCES users(user_id) ON DELETE SET NULL
)
```

`InviteStorage` exposes `create_invite(expires_at: DateTime<Utc>) -> sqlx::Result<String>` (returns the
code), `use_invite(code, user_id) -> Result<(), UseInviteError>` (validates and atomically marks
used), and `list_invites() -> sqlx::Result<Vec<InviteRecord>>`. `UseInviteError` has `NotFound`,
`Expired`, and `AlreadyUsed` variants. Code generation delegates to `generate_token()`.

### Registration policy

The registration policy is stored in `site_config` under the key `site.registration_policy`. The
`RegistrationPolicy` enum has variants `Open`, `InviteOnly`, and `Closed`, with `FromStr` parsing
`"open"`, `"invite_only"`, and `"closed"`. The default when the key is absent is `Closed`.

`load_registration_policy(store: &dyn SiteConfigStorage) -> RegistrationPolicy` reads and parses
the key, returning `Closed` on absence.

The registration server function checks the policy before creating a user: `Closed` returns an
error; `InviteOnly` requires a valid invite code and calls `use_invite` atomically with user
creation; `Open` proceeds directly.

### `jaunder user create` command

`jaunder user create --username <u> --password <p> [--display-name <n>]` creates a user directly,
bypassing the registration policy. Intended for bootstrapping an initial operator account on a
closed instance.

### `jaunder user invite` command

`jaunder user invite [--expires-in <hours>]` generates an invite code with the given expiry
(default 168 hours / 7 days), prints the raw code, and exits.

### Web UI

M2 adds six new routes to the Leptos `App`:

| Path        | Component      | Auth required |
|-------------|----------------|---------------|
| `/register` | `RegisterPage` | No            |
| `/login`    | `LoginPage`    | No            |
| `/logout`   | `LogoutPage`   | Yes           |
| `/profile`  | `ProfilePage`  | Yes           |
| `/sessions` | `SessionsPage` | Yes           |
| `/invites`  | `InvitesPage`  | Yes           |

Each page is backed by server functions that own all logic; components are thin and render the
data they receive. The invite code field on `/register` is shown only when the registration policy
is `invite_only`; a `get_registration_policy` server function provides this to the component.
`InvitesPage` is only reachable when the policy is `invite_only`; visiting it under any other
policy returns a 404.

The web UI normalises uppercase characters to lowercase as the user types in the username field.
The `register` and `login` server functions also call `.to_lowercase()` on the raw username input
before parsing it via `Username::from_str`, so server-side validation is authoritative regardless
of client behaviour.
