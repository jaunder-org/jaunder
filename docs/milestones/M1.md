# Milestone 1: Deployment bootstrap

## Scope

### CLI subcommand structure

The `jaunder` binary currently starts the server unconditionally. M1 introduces a proper CLI with two subcommands:

- `jaunder init` — creates the storage directory layout, creates the database, and applies the initial schema migrations. All guidance is in the command's help text.
- `jaunder serve` — starts the HTTP server using the bootstrap config.

### Bootstrap config

The minimum configuration needed to locate the database, start the server, and store media. Two values, each readable from a CLI flag or a `JAUNDER_*` environment variable (flag takes precedence). Each flag appears only on the subcommands that use it; a shared `StorageArgs` struct is flattened in to avoid repetition:

| Value            | CLI flag         | Env var                  | Default          | Subcommands      |
|------------------|------------------|--------------------------|------------------|------------------|
| Storage path     | `--storage-path` | `JAUNDER_STORAGE_PATH`   | `./data`         | `init`, `serve`  |
| Bind address     | `--bind`         | `JAUNDER_BIND`           | `127.0.0.1:3000` | `serve`          |

The storage path is the root of all local data: the SQLite database file lives directly in it, the media subdirectory under `media/`, and the backup subdirectory under `backups/`.

### Storage directory initialization

`jaunder init` creates the storage path and its required subdirectories (`media/`, `backups/`). If the storage path already exists and contains an initialized database, `jaunder init` exits with an error. The `--skip-if-exists` flag suppresses this error and exits successfully, allowing `jaunder init --skip-if-exists` to be used safely in scripts and container entrypoints.

### Database creation and schema migrations

`jaunder init` creates the SQLite database and applies the initial schema. The initial schema includes the `site_config` table (the runtime configuration store used by all subsequent milestones). Migrations are managed with `sqlx-cli`'s offline migration support so they are embedded in the binary and applied automatically.

### Runtime config table

The `site_config` table stores key-value pairs used by all runtime configuration. Schema:

```
site_config(key TEXT PRIMARY KEY, value TEXT NOT NULL)
```

Keys are namespaced: `site.*` for operator-set instance-wide settings, `user.<username>.*` for per-user preferences. The table is empty after `init`; rows are inserted as config is set via CLI or the web interface.

### `jaunder serve` wires bootstrap config

`jaunder serve` reads the bootstrap config (bind address and storage path), opens the database, and starts the HTTP server. The bind address from bootstrap config replaces the current hard-coded Leptos config value. If the database does not exist at the configured storage path, `serve` exits with a clear error directing the user to run `jaunder init` first.

## Steps

### Step 1: CLI argument parsing

1. [x] Add `clap` to `server/Cargo.toml`.
2. [x] Define a `StorageArgs` struct with `--storage-path` / `JAUNDER_STORAGE_PATH` that can be `#[command(flatten)]`ed into any subcommand that needs it.
3. [x] Define `Commands` enum: `Init` (flattens `StorageArgs`, adds `--skip-if-exists`) and `Serve` (flattens `StorageArgs`, adds `--bind` / `JAUNDER_BIND`).
4. [x] Parse bootstrap config: resolve CLI flag → env var → default for each value.
5. [x] Unit tests covering precedence (flag beats env var beats default) for both values.

### Step 2: Storage initialization

1. [ ] Implement `init_storage(path: &Path)` that creates the storage root, `media/`, and `backups/` subdirectories; idempotent.
2. [ ] Unit tests: new path is created correctly; existing path is left unchanged.

### Step 3: Database and migrations

1. [ ] Add `sqlx` (with `sqlite` and `migrate` features) and `sqlx-cli` to the workspace.
2. [ ] Create `migrations/` directory; add initial migration creating the `site_config` table.
3. [ ] Implement `open_database(storage_path: &Path) -> sqlx::SqlitePool` that opens (or creates) the database file and runs pending migrations.
4. [ ] Integration test: `open_database` on a fresh temp dir produces a pool with `site_config` accessible.
5. [ ] Integration test: calling `open_database` a second time on an already-migrated database succeeds without error.

### Step 4: `jaunder init` command

1. [ ] Implement `cmd_init(config: &BootstrapConfig)` that calls `init_storage` then `open_database`.
2. [ ] Print a success message confirming the storage path and database location.
3. [ ] Integration test: `cmd_init` on a fresh temp dir creates the expected directory structure and a valid database.
4. [ ] Integration test: running `cmd_init` a second time on an already-initialized path returns an error.
5. [ ] Integration test: running `cmd_init --skip-if-exists` on an already-initialized path succeeds without error.

### Step 5: `jaunder serve` command

1. [ ] Implement `cmd_serve(config: &BootstrapConfig)` that calls `open_database` and starts the Axum/Leptos server bound to the configured address. If the database does not exist at the expected path, exit with a clear error message directing the user to run `jaunder init` first.
2. [ ] Pass the database pool into the router so server functions can access it.
3. [ ] Replace the current hard-coded `get_configuration` bind address with the value from bootstrap config.
4. [ ] Integration test: `cmd_serve` fails with an appropriate error when the storage path has not been initialized.
5. [ ] Integration test: after `cmd_init`, `cmd_serve` binds to the configured address and responds to a simple health-check request.

### Step 6: Verify

1. [ ] `cargo build` succeeds.
2. [ ] `cargo nextest run` passes.
3. [ ] `cargo clippy -- -D warnings` is clean.
