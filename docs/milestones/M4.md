# Milestone 4: PostgreSQL Backend

## Scope

M4 retrofits PostgreSQL support into the existing M1–M3 feature set and establishes an ongoing
parity rule: every storage-backed feature from this point forward must either work on both SQLite
and PostgreSQL before the milestone is considered complete, or explicitly document a backend-
specific implementation strategy up front.

The `AnyPool` spike in [M3.5](./M3.5.md) ruled out a single runtime-polymorphic sqlx
implementation. M4 therefore uses:

- shared storage traits in `common`
- shared higher-level behavior and mapping helpers where practical
- backend-specific storage implementations underneath (`Sqlite*Storage`, `Postgres*Storage`)
- separate embedded migration trees under `server/migrations/sqlite/` and
  `server/migrations/postgres/`

SQLite remains the default backend. PostgreSQL is selectable via `--db` / `JAUNDER_DB`. The
accepted M4 scope is M1–M3 parity only:

- bootstrap and migrations
- site config
- users and sessions
- invites and registration policy
- SMTP config loading and mail-triggering flows
- email verification and password reset

`jaunder migrate-db --to <url>` remains part of the PostgreSQL story, but it is deferred until the
core dual-backend runtime support is stable.

## Architecture Decisions

### Backend selection at the boundary

`DbConnectOptions` expands from SQLite-only to an enum with SQLite and PostgreSQL variants. Parsing
still happens at the CLI boundary, so unsupported or malformed URLs fail before application logic
runs.

### AppState remains trait-oriented

`AppState` continues to expose trait objects:

```rust
AppState {
    site_config: Arc<dyn SiteConfigStorage>,
    users: Arc<dyn UserStorage>,
    sessions: Arc<dyn SessionStorage>,
    invites: Arc<dyn InviteStorage>,
    atomic: Arc<dyn AtomicOps>,
    email_verifications: Arc<dyn EmailVerificationStorage>,
    password_resets: Arc<dyn PasswordResetStorage>,
    mailer: Arc<dyn MailSender>,
}
```

Backend choice is resolved inside `open_database` / `open_existing_database`, which instantiate the
appropriate concrete storage implementations.

### Shared logic vs duplicated logic

Do not force all SQL through a common query layer. Use sharing only where it actually reduces
maintenance risk:

- share trait definitions and error enums
- share row-to-domain mapping helpers when the Rust-side construction is identical
- share password hashing, token hashing, and other non-SQL helpers
- keep SQL text, pool types, transaction types, and migration wiring backend-specific

### Native types are allowed

PostgreSQL should use native PostgreSQL types where they add value:

- `TIMESTAMPTZ` for timestamps
- `BOOLEAN` where boolean semantics are natural
- `BIGINT GENERATED ... AS IDENTITY` for synthetic integer keys

SQLite keeps its existing storage forms unless a different representation is required for parity.
Do not degrade PostgreSQL to text-only or least-common-denominator schemas just to imitate SQLite.

### Parity rule for future milestones

From M4 onward, any milestone that adds new persisted state must include:

1. a SQLite migration
2. a PostgreSQL migration
3. SQLite storage implementation changes
4. PostgreSQL storage implementation changes
5. tests covering both backends or backend-specific tests where behavior differs

## Steps

After each step verify:

1. `cargo build` succeeds.
2. relevant targeted tests pass for SQLite.
3. relevant targeted tests pass for PostgreSQL.
4. `cargo clippy -- -D warnings` is clean.

Wait for user review, commit when the user says that the review is complete and successful.

### Step 1: Runtime backend plumbing

1. Expand `DbConnectOptions` to parse both `sqlite:` and `postgres://` / `postgresql://` URLs.
2. Add a PostgreSQL connect-options branch in `server/src/storage/mod.rs`.
3. Split database initialization into backend-specific helpers rather than trying to hide pool
   types behind a single generic initializer.
4. Update `open_database` and `open_existing_database` to instantiate either SQLite or PostgreSQL
   storage implementations.
5. Update CLI help text and tests to reflect that PostgreSQL is now supported.
6. Add integration tests covering:
   - PostgreSQL URL parse success
   - unsupported URL rejection still works
   - `open_database` succeeds against the PostgreSQL test VM/database

### Step 2: Migration selection and bootstrap parity

1. Keep the embedded migration split introduced in M3.5:
   - `sqlx::migrate!("./migrations/sqlite")`
   - `sqlx::migrate!("./migrations/postgres")`
2. Runtime-select the migrator based on `DbConnectOptions`.
3. Ensure `jaunder init` creates storage directories regardless of backend, but only uses
   `create_if_missing` semantics for SQLite.
4. Ensure PostgreSQL bootstrap does not assume database creation inside the app if the target DB
   does not exist; fail clearly when the configured database is missing or unreachable.
5. [x] Add a separate PostgreSQL bootstrap command, `jaunder create-pg-db`, which
   uses `--bootstrap-db` plus `--app-db` to create the application role and
   database before `jaunder init` runs normal migrations.
6. [ ] Add tests for:
   - SQLite init remains unchanged
   - PostgreSQL init against an existing empty DB runs migrations successfully
   - PostgreSQL serve/open-existing on an unmigrated DB applies migrations


Current note: PostgreSQL role/database provisioning is separate admin work and
is handled by `jaunder create-pg-db`, not by `jaunder init`.

### Step 3: PostgreSQL storage implementations for M1–M3

Implement PostgreSQL equivalents for all current storage traits:

1. `PostgresSiteConfigStorage`
2. `PostgresUserStorage`
3. `PostgresSessionStorage`
4. `PostgresInviteStorage`
5. `PostgresAtomicOps`
6. `PostgresEmailVerificationStorage`
7. `PostgresPasswordResetStorage`

Requirements:

1. Keep query semantics aligned with current SQLite behavior.
2. Preserve existing user-visible errors and auth behavior.
3. Use native PostgreSQL types and SQL where appropriate instead of emulating SQLite schemas.
4. Keep transaction boundaries aligned with the SQLite implementation, especially for:
   - `create_user_with_invite`
   - session auth / `last_used_at`
   - email verification claim flow
   - password reset confirmation flow

### Step 4: Extract shared non-SQL helpers

Refactor only the shared pieces that genuinely reduce duplication between SQLite and PostgreSQL:

1. row-to-domain constructors where only the SQL row tuple differs minimally
2. password-hash creation and verification helpers
3. token-hash helpers already shared in `auth.rs`
4. small utility functions for converting backend row values into domain types

Do not introduce generic abstractions over pool/transaction types unless they reduce net
complexity.

### Step 5: Test matrix for parity

Add and organize tests so parity becomes enforceable rather than aspirational.

1. Keep current SQLite integration tests.
2. Add PostgreSQL-targeted integration tests for the same behaviors covered by current storage
   tests, at least for:
   - site config round-trips
   - user creation and duplicate username rejection
   - password authentication
   - session creation/authentication/revocation
   - invite creation/consumption
   - email verification request/use
   - password reset request/use/confirm
3. [x] Prefer shared test helpers that can run against either backend when this is straightforward.
4. [x] Use the PostgreSQL VM as the test environment until a lighter-weight local harness exists.
5. [ ] Document which tests must be run with one test thread or isolated databases to avoid backend
   cross-talk.

### Step 6: Operational integration

1. [x] Update `flake.nix` and development tooling so PostgreSQL-backed development is first-class.
2. [ ] Add documented commands for starting the PostgreSQL VM and pointing `JAUNDER_DB` at it.
3. [x] Remove SQLite-specific assumptions from scripts/tests where possible.

4. Keep existing SQLite paths intact for default local development and CI where appropriate.

### Step 7: Forward-compatibility rules for future milestones

Before closing M4, document the rules future milestones must follow:

1. new persisted features must ship both migration trees in the same change
2. storage trait changes must be implemented on both backends before merge
3. new tests must either run on both backends or state clearly why one backend is intentionally
   deferred
4. backend-specific optimizations are allowed, but behavior-visible divergence must be explicit

## Out of Scope

The following are intentionally not required to complete M4:

- full-text search parity for future content/search milestones
- migrating an existing SQLite instance into PostgreSQL or back again
- performance tuning beyond correctness and basic schema/index sanity
- replacing every current SQLite-focused test with a fully generic test harness

Those can follow once stable dual-backend support exists for the current feature set.
