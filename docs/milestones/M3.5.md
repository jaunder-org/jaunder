# Milestone 3.5: Testing for PostgreSQL Backend

## Scope

M20 adds PostgreSQL as a second database backend selectable at runtime via the `--db` flag or
`JAUNDER_DB` environment variable. SQLite remains the default. All application features work on
both backends. A `jaunder migrate-db --to <url>` command transfers data between backends without
data loss.

The implementation strategy is to replace `SqlitePool` throughout `server/src/storage/` with
sqlx's `AnyPool` — a runtime-polymorphic pool that routes to the correct driver based on the
connection URL — keeping a single set of storage implementations shared by both backends.

Before implementing M20, a focused spike is required to validate that `AnyPool` is actually
viable for this codebase. The spike results determine whether the single-implementation approach
is sound or whether a different strategy is needed.

---

## Step 1: AnyPool Feasibility Spike

This step validates the four concrete risks identified during M3 code review:

1. `DateTime<Utc>` decoding through `AnyPool`
2. Migration divergence strategy
3. `bool` type mapping
4. Remaining SQL compatibility

The spike is conducted on a throwaway branch. Its only artifacts are the findings documented
back in this file under each step below. No spike code is merged.

After each sub-step, document the finding (pass / workaround required / blocker) in this file
before moving to the next.


### Step 1.2: `DateTime<Utc>` round-trip through `AnyPool`

**Risk:** sqlx's `Any` backend represents all column values as one of a fixed set of
`AnyValueKind` variants: `Null`, `Bool`, `SmallInt`, `Integer`, `BigInt`, `Real`, `Double`,
`Text`, `Blob`. There is no native timestamp variant. It is not clear from documentation
whether `DateTime<Utc>` implements `Decode<Any>` for both a SQLite `TEXT` column and a
PostgreSQL `TIMESTAMPTZ` column, or whether PostgreSQL timestamps are surfaced as `Text`
(parseable as RFC 3339) or as some other kind that `DateTime<Utc>` cannot decode.

**Test:** Write a single test function that:

1. Opens an `AnyPool` pointed at SQLite (in-memory) and one pointed at a local PostgreSQL
   database.
2. Creates a minimal table on each:
   - SQLite: `CREATE TABLE t (ts TEXT NOT NULL, ts_null TEXT)`
   - PostgreSQL: `CREATE TABLE t (ts TIMESTAMPTZ NOT NULL, ts_null TIMESTAMPTZ)`
3. Inserts a `DateTime<Utc>` value via `.bind(now)` on each pool.
4. Reads it back via `query_as::<_, (DateTime<Utc>, Option<DateTime<Utc>>)>` on each pool.
5. Asserts the round-tripped value equals the original (within one-second tolerance to
   account for storage precision differences).
6. Also tests `Option<DateTime<Utc>>` with a NULL value to confirm `None` decodes correctly.

**Success criterion:** All assertions pass on both pools with no changes to query code.

**Fallback if it fails:** Two options to evaluate:
- **Cast to text in queries:** Append `::text` to each timestamp column reference in
  PostgreSQL queries (e.g. `SELECT ts::text FROM t`). This forces the value to arrive as
  `AnyValueKind::Text` and makes `DateTime<Utc>` parsing unambiguous. Queries become
  database-specific, but only the column reference syntax changes — not the Rust decode code.
- **Store timestamps as `TEXT` in PostgreSQL:** Use `TEXT NOT NULL` in the PostgreSQL
  migrations and bind/decode identically to SQLite. Non-idiomatic for PostgreSQL but
  eliminates any type ambiguity through `AnyPool`. Loses native PostgreSQL date range queries
  and index efficiency.

**Record finding here:** _TBD_

### Step 1.3: `bool` type mapping through `AnyPool`

**Risk:** `email_verified` is stored as `INTEGER` (0/1) in SQLite and would naturally be
`BOOLEAN` in PostgreSQL. It is bound and decoded as Rust `bool` throughout. Whether
`AnyPool` correctly decodes a PostgreSQL `BOOLEAN` column as `bool` (rather than as
`AnyValueKind::Integer`) is not guaranteed by documentation.

**Test:** Write a test function that:

1. Opens `AnyPool` on SQLite (in-memory) and PostgreSQL.
2. Creates `CREATE TABLE t (flag INTEGER NOT NULL)` on SQLite and
   `CREATE TABLE t (flag BOOLEAN NOT NULL)` on PostgreSQL.
3. Inserts `true` and `false` via `.bind(value: bool)`.
4. Reads back via `query_as::<_, (bool,)>` and asserts correct values.
5. Also confirm that binding `false` and reading it back as `bool` does not silently decode
   as `0i64` and fail the `bool` decode.

**Success criterion:** Both `true` and `false` round-trip correctly on both pools.

**Fallback if it fails:** Store `email_verified` as `SMALLINT` (0/1) in PostgreSQL migrations
and decode as `i16` or `i32` in the query, converting to `bool` manually before constructing
`UserRecord`. This is contained to `user_record_from_row` and the `authenticate` query.

**Record finding here:** _TBD_

### Step 1.4: `RETURNING` clause through `AnyPool`

**Risk:** The codebase uses `INSERT ... RETURNING` and `UPDATE ... WHERE ... RETURNING`
extensively — both with `query_scalar` (single column) and `query_as` (tuple). These are
standard SQL supported by both SQLite ≥ 3.35 and PostgreSQL, but `AnyPool` may not pass
them through correctly when combined with `fetch_optional`.

**Test:** Write test functions covering the three patterns used in the codebase:

1. `INSERT ... RETURNING id` via `query_scalar::<_, i64>(...).fetch_one(&pool)` — used in
   `create_user` and `create_user_with_invite`.
2. `UPDATE ... WHERE ... RETURNING col1, col2` via
   `query_as::<_, (T1, T2)>(...).fetch_optional(&pool)` — used in `use_email_verification`
   and `use_password_reset`. Critically, test both the "row matched" (returns `Some`) and
   "row not matched" (returns `None`) cases, since the correctness of the token-claim
   pattern depends on `fetch_optional` returning `None` when the `WHERE` clause excludes the
   row.
3. `INSERT ... RETURNING id` inside a transaction (`&mut *tx`) via `fetch_one` — used in
   `create_user_with_invite`.

**Success criterion:** All three patterns produce correct results on both SQLite and PostgreSQL
`AnyPool` instances.

**Fallback if it fails:** None expected — `RETURNING` is well-supported — but if needed, the
affected queries can be rewritten as a separate `INSERT`/`UPDATE` followed by a `SELECT
last_insert_rowid()` (SQLite) or `currval` (PostgreSQL). This would require database-specific
query paths, which defeats the purpose of `AnyPool`.

**Record finding here:** _TBD_

### Step 1.5: `is_unique_violation()` through `AnyDatabaseError`

**Risk:** Unique-constraint errors are caught via `Err(sqlx::Error::Database(e)) if
e.is_unique_violation()`. This works because both `SqliteError` and `PgDatabaseError`
implement the `DatabaseError` trait and report the correct SQLState / error code. With
`AnyPool`, the underlying error is wrapped in `AnyDatabaseError`. Whether `is_unique_violation()`
is correctly delegated through that wrapper is worth confirming.

**Test:** Write a test function that:

1. Creates a table with a `UNIQUE` constraint on both SQLite and PostgreSQL `AnyPool`
   instances.
2. Inserts a row successfully, then attempts to insert a duplicate.
3. Asserts that `e.is_unique_violation()` returns `true` on the resulting error.

**Success criterion:** `is_unique_violation()` returns `true` on both pools.

**Fallback if it fails:** Match on the error message string or SQLState code directly.
Unpleasant but isolated to the two call sites (`create_user` and `create_user_with_invite`).

**Record finding here:** _TBD_

### Step 1.6: Migration strategy selection

**Risk:** The current code uses `sqlx::migrate!("./migrations")`, a compile-time macro that
embeds a single directory of migration files. SQLite and PostgreSQL require different schemas
(column types for primary keys, timestamps, and booleans differ), so a single migration
directory cannot serve both backends. The strategy for managing two sets of migrations must
be chosen before M20 implementation begins.

**Evaluate two approaches:**

**Option A — Two embedded migration sets, runtime-selected:**
```
server/migrations/sqlite/0001_...sql
server/migrations/postgres/0001_...sql
```
In `init_pool`, select which migrator to run based on the `DbConnectOptions` variant:
```rust
match opts {
    DbConnectOptions::Sqlite(_) => sqlx::migrate!("./migrations/sqlite").run(&pool).await?,
    DbConnectOptions::Postgres(_) => sqlx::migrate!("./migrations/postgres").run(&pool).await?,
}
```
Both sets are compiled into the binary. Migration files are verified at compile time.
Tests continue to use the compile-time macro.

**Option B — Runtime filesystem migration loading:**
```rust
let dir = match opts { Sqlite => "./migrations/sqlite", Postgres => "./migrations/postgres" };
sqlx::migrate::Migrator::new(std::path::Path::new(dir)).await?.run(&pool).await?;
```
Only the selected migrations run; the other set is not embedded. Loses compile-time
verification and requires the migration files to be present on the filesystem at runtime
(complicates single-binary deployment).

**Test:** Prototype Option A in the spike branch:

1. Move existing `server/migrations/` files into `server/migrations/sqlite/`.
2. Write PostgreSQL equivalents in `server/migrations/postgres/` covering only M1–M3 tables
   (sufficient for the spike).
3. Confirm `sqlx::migrate!("./migrations/sqlite")` still compiles and runs correctly in
   existing tests.
4. Confirm `sqlx::migrate!("./migrations/postgres")` compiles and runs against the
   local PostgreSQL instance.
5. Confirm that the test helper `sqlx::migrate!("./migrations/sqlite").run(&pool)` in
   `server/tests/storage.rs` continues to work without modification (since tests always use
   SQLite).

**Success criterion:** Option A works end-to-end. If it does, record it as the chosen
strategy. If compile-time path limitations prevent it, evaluate Option B.

### Step 1.7: Evaluate spike results and record go/no-go

After completing steps 1.2–1.6, review all findings together:

1. [ ] If all tests passed without workarounds: record "AnyPool is viable as described;
   proceed with single-implementation approach."

   before beginning Step 2.
