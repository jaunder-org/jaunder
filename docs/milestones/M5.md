# Milestone 5: Simple Blog

## Scope

M5 adds the core content creation and reading features: users can write posts (in Markdown or
Org-mode), publish or save them as drafts, edit and soft-delete them, and read them back via
permalink, per-user timeline, or the main timeline. This builds on the auth layer from M2 and
maintains backend parity per the M4 rule.

## Architecture Decisions

### Post model

A post has: id, user_id, title, slug, body, format (markdown or org), rendered_html, created_at,
updated_at, published_at, and deleted_at.

- `published_at` is null for drafts.
- `deleted_at` is null for live posts; soft-deleted posts are excluded from all public queries.
- `rendered_html` stores the server-side rendered output, generated at write time.

### Slug generation

Slugs are auto-generated from the title but can be overridden by the author. Uniqueness is enforced
per user per date (year/month/day). While a post is in draft, the slug can change freely. Once
published, the slug is frozen to preserve permalinks.

### Content rendering

Markdown and Org-mode are rendered to HTML server-side at write time. The rendered output is stored
in the `rendered_html` column alongside the source body. Editing a post re-renders and updates the
stored HTML.

### URL structure

- Permalink: `/~:username/:year/:month/:day/:slug`
- Per-user timeline: `/~:username/`
- Main timeline: `/` (local timeline when unauthenticated, user's own posts when authenticated)

### Pagination

All timelines use cursor-based pagination, reverse chronological by `created_at`, 50 entries per
page. The UI should present smooth/infinite scrolling rather than explicit page links.

### Drafts

Drafts are posts with a null `published_at`. They are visible only to their author and clearly
marked as drafts in the UI. A draft must have a title.

### Soft delete

Deletion sets `deleted_at` rather than removing the row. Soft-deleted posts are excluded from all
public views and return 404 at their permalink.

### Tags

Posts can have zero or more tags. Tags are case-insensitive for querying/indexing but case-preserving
for display (so #Ruby and #ruby refer to the same tag, but each post displays the tag as the author
wrote it).

- `tags` table stores canonical tag slugs (lowercase, alphanumeric + hyphens, same format as post slugs).
- `post_tags` join table associates posts with tags, storing the case-sensitive display form each author used.
- Query endpoints `/tags/:tag` and `/~:username/tags/:tag` match case-insensitively and return posts in
  cursor-paginated, reverse chronological order (50 per page).
- Tags can be edited when a post is edited, even after publication.

## Out of Scope

The following are intentionally not required to complete M5:

- Revision history UI and API (deferred to M15; schema support is included in M5)
- Comments or replies (deferred to M15)
- Media attachments in posts (deferred to M7)
- RSS/Atom/JSON Feed generation (deferred to M8)
- ActivityPub federation of posts (deferred to M12)
- Full-text search over posts (deferred to M11)
- Tag discovery (tag cloud, trending tags, tag autocomplete; unscheduled, to be prioritized in roadmap)
