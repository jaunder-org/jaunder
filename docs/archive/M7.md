# Milestone 7: Media Handling

## Scope

M7 adds user media upload with local serving, and a caching system for remote media
attachments consumed from external sources. User-uploaded media is served publicly via
content-addressed URLs. Remote media caching protects against link rot and is configurable
per-user (eager, lazy, or off). Both systems share a content-addressed filesystem layout
with hard-link deduplication.

## Architecture Decisions

### Storage approach: hybrid

Media is stored content-addressed on disk (SHA256), with a database table tracking ownership
and metadata. The database is the source of truth for what media exists; the filesystem is
the content store. Hard-links deduplicate identical content across users and across the
upload/cached trees.

### Filesystem layout

```
<storage_path>/media/
  upload/
    <4-hex-prefix>/
      <full-sha256>/
        <original-filename>
  cached/
    <4-hex-prefix>/
      <full-sha256>/
        <original-filename>
  tmp/
    (in-progress uploads, cleaned on failure)
```

- The 4-hex prefix is the first 4 characters of the SHA256 hex string (65,536 sharding
  buckets).
- Each hash directory may contain multiple filenames (same content uploaded under different
  names by the same or different users). All files in a hash directory are hard-links to
  each other.
- Hard-links also cross the `upload/` and `cached/` boundaries when content matches.
- URL structure: `/media/upload/<prefix>/<hash>/<filename>` and
  `/media/cached/<prefix>/<hash>/<filename>`.
- URLs are not guessable without knowing the file content (SHA256 serves as the
  unguessable component).

### Upload flow (robustness)

1. Check `Content-Length` header against configured max; reject immediately if over.
2. Stream upload to a temp file in `<storage_path>/media/tmp/`, computing SHA256
   incrementally and tracking bytes written. **Abort and delete temp file if bytes written
   exceeds the configured max.**
3. Validate: user quota not exceeded (current usage + new file size).
4. Determine target path: `upload/<prefix>/<hash>/<filename>`.
5. If target directory already has any file: hard-link that existing file to the new
   filename (same content, guaranteed by hash). If the exact filename already exists, no
   filesystem work needed.
6. If target directory doesn't exist: create it, move temp file in.
7. **Only after filesystem write succeeds**: INSERT the DB record.
8. On DB failure: the file on disk is an orphan (harmless, can be GC'd later).
9. Clean up temp file on any failure path.

Ordering guarantees: orphaned files (on disk but no DB record) waste space but cause no
correctness issues. Orphaned DB records (row exists but file missing) would be a problem;
the file-first ordering prevents this.

### Filename sanitization

Uploaded filenames are sanitized before use in filesystem paths and URLs:
- Strip any path components (directory traversal attempts like `../../etc/passwd` become
  `passwd`).
- Replace characters unsafe for filesystems or URLs (null bytes, slashes, backslashes) with
  underscores.
- Reject empty filenames after sanitization.
- Preserve the original filename as-is in the DB `filename` column (for display purposes)
  only if it passes basic validation; otherwise store the sanitized form.

### File types and rendering

Any file type may be uploaded. An allowlist determines which types are served with
`Content-Disposition: inline` (rendered in-browser):

- Images: `image/jpeg`, `image/png`, `image/gif`, `image/webp`, `image/svg+xml`
- Audio: `audio/mpeg`, `audio/ogg`, `audio/flac`, `audio/wav`
- Video: `video/mp4`, `video/webm`
- PDF: `application/pdf`

All other types are served with `Content-Disposition: attachment; filename="<original>"`.

### Media-to-post relationship

Media is not formally linked to posts via a database join table. Users reference media in
their post body by its URL (e.g., `![photo](/media/upload/a3f2/.../photo.jpg)` in Markdown).
The relationship is implicit in the content. This avoids sync problems between a join table
and the actual post body.

When a user attempts to delete a media item, the system scans their posts for references to
that media's URL and warns if any are found. This scan happens at delete time only (not
materialized).

### Access model

All media (both uploaded and cached) is publicly accessible once stored. No authentication
is required to retrieve media. This is cache-friendly and works well with CDNs and
federation. The unguessable SHA256-based URL path provides practical obscurity.

### Serving headers

- `Cache-Control: public, max-age=31536000, immutable` (content never changes at a given
  URL)
- `ETag: <sha256>`
- `Content-Type` from DB record (fallback: extension-based detection)
- `If-None-Match` support (304 Not Modified)
- `Content-Disposition: inline` for rendered types, `attachment` for others

### Quotas

Operator-configurable via site_config:

| Key | Default | Description |
|-----|---------|-------------|
| `media.max_file_size_bytes` | `52428800` (50MB) | Per-file upload limit |
| `media.user_quota_bytes` | `1073741824` (1GB) | Per-user total storage quota |
| `media.cache_policy_default` | `off` | Default cache policy (`eager`, `lazy`, `off`) |

### Caching remote media

When Jaunder ingests content from external sources (M9: ActivityPub, M17: Feeds), remote
media URLs referenced in that content can be cached locally per the user's preference.

#### Per-user cache policy

Each user chooses one of:
- **eager**: At ingestion time, immediately download and cache all media referenced in
  incoming content for that user's subscriptions.
- **lazy**: Cache on first local access via a proxy endpoint.
- **off**: Never cache; leave remote URLs as-is in the processed form.

The operator sets a site-wide default via `media.cache_policy_default` in site_config. Users
override via their `user_config` entry (key: `media.cache_policy`).

#### Cache ingestion flow

1. Content arrives (future M9/M17 delivers it).
2. Extract media URLs from the content body.
3. Based on the subscribing user's cache policy:
   - **eager**: Enqueue download immediately.
   - **lazy**: Rewrite URLs in the processed form to point to the lazy proxy endpoint.
   - **off**: Leave URLs unchanged.
4. Download: fetch remote file, stream to temp (enforcing `media.max_file_size_bytes` with
   streaming cutoff), compute SHA256.
5. Store in `media/cached/<prefix>/<hash>/<original-filename>` (hard-link if content
   already exists in either tree).
6. Insert DB record with `source = 'cached'` and `source_url` set.
7. In the **processed/rendered form** of the content, rewrite remote media URLs to point to
   the local cached copy. The **raw payload** (original AP JSON-LD, RSS XML, etc.) is stored
   verbatim and never modified.

The raw payload is always the source of truth. If caching policy changes or cached media is
purged, the processed form can be re-derived from the raw payload.

#### Lazy proxy endpoint

```
GET /media/proxy?url=<encoded-remote-url>&user_id=<id>
```

- Requires authentication (requesting user must match `user_id`).
- On first hit: fetches remote, caches, redirects to the local cached URL.
- On subsequent hits: redirects to cached URL.
- If caching fails (remote down, too large): passes through to remote URL.

### Crate layout

Following Approach A (existing patterns):

- `common/src/storage/media.rs`: `MediaStorage` trait, `MediaRecord` struct, error types,
  media-related site_config key constants. (Part of the `common/src/storage/` module
  directory created in Step 1's refactor.)
- `common/src/storage/user_config.rs`: `UserConfigStorage` trait.
- `common/src/media.rs`: Pure functions for URL derivation, path computation from hash +
  filename + source, filename sanitization. No filesystem dependencies.
- `server/src/media.rs` (or `server/src/media/`): Upload handler, serving handler, lazy
  proxy handler, filesystem operations (streaming, SHA256, hard-linking, deduplication),
  quota enforcement.
- `server/src/storage/sqlite.rs` and `postgres.rs`: `MediaStorage` and `UserConfigStorage`
  trait implementations.
- `web/src/media.rs`: Server functions for media management (list, delete, usage).
- `web/src/pages/media.rs`: Media management page component.

## Data Model

### `media` table

```sql
CREATE TABLE media (
    user_id       BIGINT NOT NULL REFERENCES users(user_id),
    sha256        TEXT NOT NULL,
    filename      TEXT NOT NULL,
    source        TEXT NOT NULL CHECK (source IN ('upload', 'cached')),
    content_type  TEXT NOT NULL,
    size_bytes    BIGINT NOT NULL,
    source_url    TEXT,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (user_id, sha256, filename, source)
);
```

- `source_url`: NULL for uploads, populated for cached media (the original remote URL).
- Two users uploading the same file get separate rows (same sha256, different user_id) but
  share disk space via hard-links.

### User config table

Per-user preferences are stored in a dedicated `user_config` table keyed by `(user_id, key)`:

```sql
CREATE TABLE user_config (
    user_id  BIGINT NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
    key      TEXT NOT NULL,
    value    TEXT NOT NULL,
    PRIMARY KEY (user_id, key)
);
```

This is immune to username changes, provides clean indexed lookups, and cascades on user
deletion. M7 introduces this table; future milestones will use it for other per-user
preferences.

| Key | Values | Description |
|-----|--------|-------------|
| `media.cache_policy` | `eager` / `lazy` / `off` | Per-user cache policy override |

## HTTP Endpoints

### Upload (Axum handler, not a server function)

```
POST /media/upload
Content-Type: multipart/form-data
Authorization: (session cookie)

Response 201:
{
  "sha256": "a3f2...",
  "filename": "photo.jpg",
  "content_type": "image/jpeg",
  "size_bytes": 245832,
  "url": "/media/upload/a3f2/a3f2.../photo.jpg"
}
```

### Serving (Axum handler)

```
GET /media/upload/<prefix>/<hash>/<filename>
GET /media/cached/<prefix>/<hash>/<filename>
```

Public, no auth. Returns the file with caching headers.

### Lazy proxy (Axum handler)

```
GET /media/proxy?url=<encoded>&user_id=<id>
```

Authenticated. Fetches, caches, and redirects (or passes through on failure).

### Management (server functions)

| Endpoint | Description |
|----------|-------------|
| `list_my_media` | Paginated list of user's media (size, date, URL, source) |
| `delete_media` | Delete a record + unlink file (warns if referenced in posts) |
| `media_usage` | Current usage vs quota |

## Future considerations (not in M7 scope)

- Thumbnail generation (hash directory structure accommodates derivatives naturally)

## Out of Scope

- Reverse-proxy file serving offload (future optimization)
- CDN configuration
- Image/video transcoding or thumbnail generation
- Avatar upload UI (can be added later using this infrastructure)
- Background GC of orphaned files
- Materialized post-to-media reference index
