# Spec — `PostBody` / `PostTitle` newtypes (issue #402)

- **Issue:** [#402](https://github.com/jaunder-org/jaunder/issues/402) — "types:
  PostBody/PostTitle newtypes (deferred; pairs with RenderedHtml)"
- **Milestone:** Domain-value type safety (newtypes)
- **ADRs:** ADR-0063 (domain-value newtype convention); pairs with
  `RenderedHtml` (#398). ADR-0065 (client-side validation) is **not** engaged —
  see §Non-goals.
- **Date:** 2026-07-15

## Problem

A post's raw source `body` and `title` cross the codebase as bare `String`
(storage records/inputs, the `#[server]` create/edit boundary, DTOs, the render
pipeline, feed/atompub mapping). The strongest justification is **transposition
safety**: raw source must be distinct from rendered output so a `PostBody` can
never be handed where the already-shipped `RenderedHtml` (#398) is expected.
Secondary payoff is consistency with the rest of the newtype set. Per ADR-0063
these are its explicitly lower-priority tier (weak/consistency justification),
which is why #402 is the deferred member.

## Decisions

### D1 — Both types are **infallible** (no length bound, no rejection)

`PostBody` and `PostTitle` impose **no length bound** and reject nothing. A body
cap is a guaranteed future foot-gun (someone will always want a larger post),
and the issue's real payoff is the `RenderedHtml` transposition guard, not
validation. Construction therefore cannot fail: the door is `From<String>`, not
a `Result`-returning `TryFrom`/`FromStr`.

Consequence: because construction is infallible, typing the `body` wire arg as
`PostBody` has **no decode-failure mode**, so no client-side pre-validation is
required (ADR-0065 is not engaged).

### D2 — `PostTitle` normalizes by **trimming**; `PostBody` is a pure wrapper

"Infallible" constrains _rejection_, not _normalization_. `PostTitle`'s
`From<String>` trims leading/trailing whitespace (preserving case and internal
whitespace); `PostBody`'s `From<String>` wraps verbatim (trimming a body would
eat meaningful leading/trailing content and formatting). Any collapse of an
empty-after-trim title to `None` is the responsibility of the field/metadata
layer (`derive_post_metadata`), not the type.

### D3 — Teach `StrNewtype` an `infallible` mode (macros crate)

The `StrNewtype` derive currently routes all construction through a hand-written
`FromStr` (emitting `TryFrom<String>` + a `FromStr`-routed `Deserialize`).
ADR-0063's trailer already promises "`TryFrom<String>` **(or `From<String>` when
infallible)**", but the infallible half was never implemented because nothing
needed it. `PostBody`/`PostTitle` are the first infallible str-newtypes, so this
issue implements it (keeping the macro variant in-scope per the "don't
over-fragment" convention).

**Infallible-mode contract** —
`#[derive(StrNewtype)] #[str_newtype(infallible)]` on `struct X(String)`:

- **Generated by the derive:** `Display`, `AsRef<str>`, `Borrow<str>`,
  `Deref<Target = str>`, `From<X> for String`, `PartialEq<str>`,
  `PartialEq<&str>`, a borrowing `Serialize` (`serialize_str(&self.0)`), and a
  `Deserialize` that deserializes a `String` and routes it through the type's
  own `From<String>` (so wire values are normalized identically to in-process
  construction).
- **NOT generated** (so it can normalize/reject-nothing per type):
  `From<String>` — the type author hand-writes it. This mirrors the fallible
  mode, where the author hand-writes `FromStr`.
- **Omitted entirely:** `FromStr`, `TryFrom<String>` (there is no fallible
  door).
- **`parse_opts`** accepts `infallible` and rejects it combined with `secret` or
  `serde` (a spanned `compile_error!`), since those select unrelated surfaces.

### D4 — `render()` and metadata derivation speak the newtypes

`render(body: &str, …) -> RenderedHtml` becomes
`render(body: &PostBody, …) -> RenderedHtml`, making both ends of the pipeline
distinct types (the issue's stated goal). `derive_post_metadata`'s output
(`DerivedPostMetadata.title`, `render.rs:135`) becomes `Option<PostTitle>`.
Because the trailer provides `Deref<str>`, internal code that needs `&str` still
works by deref.

`render()` is called only in `storage/src/post_service.rs`. For the org format
its input is not the raw body but the `String` from `canonicalize_org_body`;
that canonicalized string is still raw post source, so it is **re-wrapped**
(`.into()`, infallible) into a `PostBody` before `render()`. (Alternative
considered and rejected: leaving `render(body: &str)` and passing `&post_body`
via `Deref` — it weakens the "input is `PostBody`" end of the pipeline the issue
asks for.)

### D5 — Thread the newtypes through every body/title site (one PR)

Complete the mechanical sweep in this change — no deferred tails:

- **storage records/inputs:** `PostRecord.body`, `PostRevisionRecord.body`,
  `CreatePostInput.body`, `UpdatePostInput.body` → `PostBody`; the `title`
  fields (`Option<String>`) → `Option<PostTitle>`. Row-mappers convert with
  `.into()`; SQL binds use `Deref`/`AsRef`.
- **`post_service` perform-aggregates:** `PostCreation<'a>` and `PostUpdate<'a>`
  carry `body: String` and a **borrowed** `title: Option<&'a str>`. `body` →
  `PostBody`. For the borrowed title, default to keeping a borrow of the owned
  input title — `Option<&'a PostTitle>` — and rely on `Deref<str>` where only
  the text is read; the plan pins the exact form. `helpers`, `test_support`
  follow.
- **web `#[server]` boundary:** `create_post`/`update_post` `body: String` →
  `PostBody`; DTOs in `web/src/posts/mod.rs` and `web/src/posts/listing.rs`
  carrying title/body.
- **server/atompub:** the post↔entry **mapping** layer only —
  `server/src/atompub/mapping.rs`: `PostFields.title/body`, the inbound door
  `entry_to_post_fields`, and the read-out `post_to_entry`. **Excluded** (these
  are _not_ post domain values and must stay as-is): Atom-native `Text` and the
  feed-label `String`s in `common/src/atompub/entry.rs` (`CollectionMeta.title`,
  `MediaEntryMeta.title`), and `common/src/feed/metadata.rs`'s `FeedItem.title`
  (a generic/ingested feed-item title, not our post's `PostTitle`).
- **tests** across all of the above.

## Non-goals

- **No length/format validation** and no `ValidatedInput`/`Field<T>` client-side
  validation work (D1 makes construction infallible → nothing to pre-validate;
  ADR-0065 is not engaged). The post compose/edit form is untouched except for
  the `body` argument's type, which decodes any string.
- **No change to `RenderedHtml`** or its trusted-rebuild door.
- **No separable follow-up issues** — the macro `infallible` mode is in-scope
  here, and no out-of-scope concerns surfaced.

## Acceptance criteria (observable)

1. **AC-macro-infallible.** A macros-crate unit test (via `syn::parse_quote!` on
   `expand()`, coverage-measured, modeled on the existing
   `str_newtype_secret_selects_redacting_trailer` test) asserts the infallible
   arm's generated **impl set** at the token level: `Display`, `AsRef`, `Deref`,
   `From < X > for String`, `Serialize`, `Deserialize` are **present** and
   `TryFrom`, `FromStr` are **absent**. (Runtime JSON/behavioral round-trip is
   covered by AC-3/AC-4 against the real `common` types, not here.)
2. **AC-macro-reject.** `#[str_newtype(infallible, secret)]` and
   `#[str_newtype(infallible, serde)]` fail to compile with a spanned message; a
   macros-crate unit test asserts `parse_opts` errors on each.
3. **AC-postbody-type.** `PostBody(String)` exists in `common`, is infallible
   (`From<String>` pure wrap, no `FromStr`/`TryFrom`), and serializes as a plain
   string. Unit tests: wrap round-trip, `Display`, `Deref`, serde plain-string
   round-trip, `From<PostBody> for String`.
4. **AC-posttitle-trim.** `PostTitle(String)` exists in `common`, is infallible,
   and `PostTitle::from("  Hello  World ".to_owned())` equals `PostTitle` of
   `"Hello  World"` (outer trim, inner whitespace + case preserved).
   Deserializing `"\"  x \""` yields `PostTitle` of `"x"`. Unit tests assert
   both the construction-trim and the deserialize-trim.
5. **AC-transposition.** A `PostBody` cannot be passed where a `RenderedHtml` is
   expected and vice-versa: a `compile_fail` doctest (as `RenderedHtml` uses)
   demonstrates the type error.
6. **AC-boundary.** `create_post`/`update_post` accept `PostBody` for `body`;
   the storage
   `PostRecord`/`PostRevisionRecord`/`CreatePostInput`/`UpdatePostInput` `body`
   fields are `PostBody` and `title` fields are `Option<PostTitle>`; `render`'s
   `body` parameter is `&PostBody`; the web DTOs (`PostResponse`,
   `DraftSummary`, `TimelinePostSummary`) and the atompub
   `mapping.rs::PostFields` body/title fields adopt the newtypes and the crate
   compiles. (Observable in the type signatures; existing storage/web tests
   still pass through the new types.)
7. **AC-gate.** `cargo xtask validate --no-e2e` is clean (host static + clippy +
   coverage), with no coverage regression from the new `common`/macros code.

## Risks / notes

- **Blanket-impl collision:** the derive must _not_ auto-generate `From<String>`
  in infallible mode — a hand-written `From<String>` plus the fallible mode's
  `TryFrom<String>` would collide via std's blanket
  `impl<T, U: Into<T>> TryFrom<U>`. The contract in D3 (author writes
  `From<String>`, derive omits `TryFrom`) avoids this.
- **Trim idempotence on the wire:** `PostTitle` values are trimmed at creation
  and again on deserialize; trimming is idempotent, so re-trimming a stored
  value is a no-op.
- **Coverage:** the macros crate is coverage-measured; the new infallible arm
  and its rejection paths need in-crate `parse_quote!` tests (not just usage
  from `common`).
