# Milestone 1: Deployment bootstrap

## Scope

### CLI subcommand structure

The `jaunder` binary currently starts the server unconditionally. M1 introduces a proper CLI with two subcommands:

- `jaunder init` — creates the storage directory layout, creates the database, and applies the initial schema migrations. All guidance is in the command's help text.
- `jaunder serve` — starts the HTTP server using the bootstrap config.

### Bootstrap config

The minimum configuration needed to locate the database, start the server, and store media. Each value is readable from a CLI flag or a `JAUNDER_*` environment variable (flag takes precedence). Each flag appears only on the subcommands that use it; a shared `StorageArgs` struct is flattened in to avoid repetition:

| Value        | CLI flag         | Env var                | Default                              | Subcommands     |
|--------------|------------------|------------------------|--------------------------------------|-----------------|
| Storage path | `--storage-path` | `JAUNDER_STORAGE_PATH` | `./data`                             | `init`, `serve` |
| Database URL | `--db`           | `JAUNDER_DB`           | `sqlite:./data/jaunder.db`           | `init`, `serve` |
| Bind address | `--bind`         | `JAUNDER_BIND`         | `127.0.0.1:3000`                     | `serve`         |

The storage path is the root of all local file data: the media subdirectory lives under `media/` and the backup subdirectory under `backups/`. The database URL encodes both the engine and connection parameters; the only engine supported in M1 is SQLite.

### Storage directory initialization

`jaunder init` creates the storage path and its required subdirectories (`media/`, `backups/`). If the storage path already exists, `jaunder init` exits with an error. The `--skip-if-exists` flag suppresses this error and exits successfully, allowing `jaunder init --skip-if-exists` to be used safely in scripts and container entrypoints.

### Database creation and schema migrations

`jaunder init` creates the database and applies the initial schema. The initial schema includes the `site_config` table (the runtime configuration store used by all subsequent milestones). Migrations are managed with `sqlx-cli`'s offline migration support so they are embedded in the binary and applied automatically.

### Database backend abstraction

Database access is mediated through per-subsystem traits with a `Storage` suffix (e.g., `SiteConfigStorage`). Each trait defines the async operations for that subsystem; concrete implementations (`SqliteSiteConfigStorage`, later `PostgresSiteConfigStorage`) contain all sqlx-specific code and use compile-time `query!()` macros against the backend's native pool type. Application code and server functions depend only on trait references — nothing outside the concrete implementations imports sqlx or a backend-specific type. `open_database(db_url: &str)` opens the connection, runs migrations, and returns the concrete implementation hidden behind the trait. SQLite is the only supported backend in M1; PostgreSQL is added in M20.

### Runtime config table

The `site_config` table stores key-value pairs used by all runtime configuration. Schema:

```
site_config(key TEXT PRIMARY KEY, value TEXT NOT NULL)
```

Keys are namespaced: `site.*` for operator-set instance-wide settings, `user.<username>.*` for per-user preferences. The table is empty after `init`; rows are inserted as config is set via CLI or the web interface.

### `jaunder serve` wires bootstrap config

`jaunder serve` reads the bootstrap config (bind address, storage path, and database URL), opens the database, and starts the HTTP server. The bind address from bootstrap config replaces the current hard-coded Leptos config value. If the database cannot be opened at the configured URL, `serve` exits with a clear error directing the user to run `jaunder init` first.
