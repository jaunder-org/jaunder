# Spec ‚Äî issue #459: `ContentHash` newtype for the media content hash

- Issue: [#459](https://github.com/jaunder-org/jaunder/issues/459) ‚Äî _types:
  ContentHash/Sha256 newtype for media content hash_
- Milestone: Domain-value type safety (newtypes); track 3 of 3 split from #17
- ADRs applied: ADR-0063 (domain-value newtype convention, str-trailer variant),
  ADR-0065 (typed `#[server]` wire args with client-side pre-validation)
- Prereq: `StrNewtype` derive exists (`macros/src/str_newtype.rs`, #14 closed) ‚Äî
  actionable now.

## Goal

The media content hash ‚Äî the SHA-256 hex digest that content-addresses a stored
media object ‚Äî is a bare `String`/`&str` today (`MediaRecord.sha256`,
`get_media(sha256: &str)`, `delete_media`, `find_by_hash`, the upload/serve
DTOs, the web wire DTO and server fn). Introduce a validated-hex str-newtype
`ContentHash` (ADR-0063 default fallible trailer) so an arbitrary string can no
longer be passed where a content hash is expected, and thread it **pervasively**
through every site the value appears ‚Äî storage records/traits/impls, the server
compute/serve/AtomPub paths, and the web DTO/server-fn/form path.

This is a behavior-preserving refactor: the wire form of the hash is unchanged
(a plain 64-char lowercase hex string everywhere it serializes), and no
user-visible behavior changes except the two defense-in-depth error paths noted
below.

## The type

Add to `common/src/media.rs` (cohesive with the existing
`is_valid_content_hash`, `media_path`, `media_url` helpers that already live
there):

```rust
/// A validated media content hash: exactly 64 lowercase hex chars
/// (`[0-9a-f]{64}`), the canonical `format!("{digest:x}")` form of a SHA-256
/// digest. The ADR-0063 string-newtype trailer is generated by
/// `#[derive(StrNewtype)]`; `FromStr` is the single validating chokepoint.
#[derive(Clone, Debug, PartialEq, Eq, Hash, StrNewtype)]
pub struct ContentHash(String);

/// Error returned when a string is not a canonical 64-char lowercase-hex hash.
#[derive(Debug, Error)]
#[error("content hash must be 64 lowercase hex characters ([0-9a-f]{{64}})")]
pub struct InvalidContentHash;

impl FromStr for ContentHash {
    type Err = InvalidContentHash;
    fn from_str(s: &str) -> Result<Self, Self::Err> {
        if is_valid_content_hash(s) {
            Ok(ContentHash(s.to_owned()))
        } else {
            Err(InvalidContentHash)
        }
    }
}

impl ContentHash {
    /// Wraps a hash we ourselves produced ‚Äî a freshly computed
    /// `format!("{digest:x}")` string, or a value read back from our own `sha256`
    /// column ‚Äî **without** re-validating. The single trusted-construction door
    /// (mirroring `TokenHash::from_digest`); **untrusted** input (a URL path
    /// segment, a wire arg) must go through `FromStr`/`TryFrom`, which validate.
    #[must_use]
    pub fn from_digest(digest: impl Into<String>) -> Self {
        ContentHash(digest.into())
    }
}
```

**`from_digest` door choice (`impl Into<String>` vs `[u8; 32]`) ‚Äî decided
consciously.** The adversarial review argued for `from_digest(bytes: [u8; 32])`
formatting hex internally: it would be structurally unforgeable (32 bytes ‚áí
always 64 hex) and delete the three duplicated `format!("{digest:x}")` sites,
and the `Sha256::digest(bytes).into()` ‚Üí `[u8; 32]` conversion is a proven-clean
idiom (`storage/src/backup.rs:564`). That is genuinely nicer _in isolation_.
**We do not adopt it**, for one decisive reason: the direct sibling hash newtype
`TokenHash` (landed one week ago, #458) uses exactly
`from_digest(impl Into<String>)` and uses that _same one door_ both at its
compute site (`host/src/token.rs:58,67`) **and** for its trusted DB-read rebuild
(`storage/src/helpers.rs:93`, "trusted rebuild of our own stored value").
`ContentHash` is the same shape and needs the same dual use (below). A
`[u8; 32]` door serves only the compute site and would force a _second_
String-taking door for the DB read ‚Äî two doors where the freshly established
convention has one. Consistency with the immediately-preceding sibling beats the
local optimization. Consequence honestly stated: the `impl Into<String>` door is
not self-validating, so the "no panic in `media_path`" property below is a
**contract** guarantee (every `ContentHash` reaching `media_path` is either
`FromStr`-validated to 64 hex, or `from_digest`'d from a real SHA-256 digest),
not a type-system theorem ‚Äî exactly as it is for `TokenHash`.

Design choices and their precedents:

- **Default fallible trailer** (not `secret`, not `infallible`): the value has a
  format invariant (64 lowercase hex), so `FromStr` is the validating chokepoint
  and the serde bridge rejects invalid input on the wire ‚Äî the exemplar is
  `Email` (`common/src/email.rs`).
- **`FromStr` reuses `is_valid_content_hash`** ‚Äî the predicate already exists
  and is the single source of truth for "canonical content hash".
  `is_valid_content_hash` itself **stays `&str`**: it is the validation
  primitive that _produces_ the guarantee from untrusted input, so it
  necessarily operates below the type. This is not a skipped site ‚Äî it is the
  constructor's engine.
- **Trusted `from_digest(impl Into<String>)`** ‚Äî mirrors
  `TokenHash::from_digest` (#458) exactly: a hash produced from a digest is
  well-formed by construction, so the compute path wraps without a redundant
  re-parse (and without a denied `.expect()`). This is the same two-door shape
  as `RawToken`/`TokenHash`: a validating `FromStr` for untrusted input, a
  trusted constructor for the server-side producer. `ContentHash` is **not** a
  security newtype (it is a hash, freely rendered and transmitted ‚Äî like
  `TokenHash`), so it needs no `xtask` construction gate; the open `FromStr`
  door is correct, as for `Email`.
- **Derives** `Clone, Debug, PartialEq, Eq, Hash` ‚Äî the standard set (matching
  `Email`/`TokenHash`). No `Ord` (never a Rust-side sort/map key; the composite
  PK ordering is DB-side). No `Copy` (owns a `String`).

### Test-support helper

Add to `common/src/test_support.rs`, following the established convention
(`parse_email`, `parse_audience_name`, `parse_raw_token`):

```rust
/// Parse `s` into a valid [`ContentHash`] for tests ‚Äî the single place a test
/// content-hash literal is parsed. Panics if `s` is not 64 lowercase hex chars.
#[must_use]
pub fn parse_content_hash(s: &str) -> ContentHash {
    s.parse().expect("valid test content hash")
}
```

Every `cfg(test)` site that builds a media hash value uses `parse_content_hash`
(per the "Newtype test-helper convention" ‚Äî no inline `.parse().unwrap()` or
per-module helper). Short placeholder hashes in existing tests (e.g. `"abc123"`,
`"sha256"`, the 20-char `media_path` fixtures) become realistic 64-char values.

## Pervasive threading ‚Äî the scope, site by site

The governing line: **within this issue's scope (the media content hash /
`sha256` value), use `ContentHash` at every single site the value appears.**
Sibling values that share the record but belong to other issues ‚Äî `user_id`
(#471), `filename`, `source` (already the `MediaSource` enum) ‚Äî are left exactly
as they are.

### `common/src/media.rs`

- **Add** `ContentHash` + `InvalidContentHash` + `from_digest` (above).
- `media_path(source: &str, sha256: &ContentHash, filename: &str)` ‚Äî the
  `sha256` param becomes `&ContentHash`. Internally the `sha256[..2]`/`[2..4]`
  slicing is unchanged (via `Deref<str>`). **This is a type-safety win**: the
  documented "panics on a short / non-UTF-8-boundary hash" DoS hazard is
  eliminated in practice ‚Äî every `ContentHash` that reaches `media_path` is
  either `FromStr`-validated (64 ASCII hex) or `from_digest`'d from a real
  digest (see the door-choice note above; this is a contract guarantee, matching
  `TokenHash`, not a raw-`str` caller that could pass `"a"`). Update the
  module + fn docs to reflect that the caller now hands in a validated content
  hash rather than an arbitrary `&str`. `source`/`filename` stay `&str` (out of
  scope).
- `media_url(source: &str, sha256: &ContentHash, filename: &str)` ‚Äî same; it
  delegates to `media_path`.
- `is_valid_content_hash(hash: &str)` ‚Äî **unchanged** (the validation primitive,
  see above).

### `storage/` ‚Äî record, trait, impls, dialect, row helper

- `MediaRecord.sha256: ContentHash` (field, `storage/src/media.rs`).
- `MediaStorage` trait: `get_media`, `delete_media`, `find_by_hash` take
  `sha256: &ContentHash` (parity with the sibling `source: &MediaSource` already
  in these signatures ‚Äî the trait is the storage boundary, ADR-0063 ¬ß4).
- `MediaStore` impls: SQL binds change from `.bind(sha256)` /
  `.bind(record.sha256.as_str())` to bind through `AsRef<str>`
  (`.bind(sha256.as_ref())` / `.bind(record.sha256.as_ref())`). The
  `for<'q> &'q str: Encode` bound is satisfied unchanged.
- `MediaDialect::delete_media_row(..., sha256: &ContentHash, ...)` ‚Äî threaded
  here too (full within-scope pervasiveness). The **sibling `source: &str`** on
  this same helper stays `&str` because `MediaSource ‚Üí str` flattening is
  #459-out-of-scope, not because the layer is exempt. `delete_media` calls it
  with `sha256` directly. The two backend impls
  (`storage/src/{sqlite,postgres}/media.rs`) bind `.bind(sha256.as_ref())`.
- `storage/src/helpers.rs`: `MediaRow` tuple keeps element 2 as `String` (it is
  the raw sqlx column projection of primitives; the domain type appears in the
  record ‚Äî exactly as `source` is a `String` in the tuple but `MediaSource` in
  the record). `media_record_from_row` **trusted-wraps on read**:
  `sha256: ContentHash::from_digest(sha256)`. This mirrors the direct peer
  `TokenHash::from_digest(token_hash)` at `helpers.rs:93` ("trusted rebuild of
  our own stored value") ‚Äî the `sha256` column is written **only** by our own
  compute path, so re-validating it is not the peer-consistent choice and would
  (a) diverge from how `TokenHash` is rebuilt two functions away, (b) add a new
  fallible `Decode` branch needing its own coverage, and (c) break the existing
  `media_record_from_row_{accepts,rejects}_*` tests, which pass a short
  placeholder `"sha256"` in the hash slot (a validate-on-read would make
  `_rejects_invalid_source` fail on the _hash_ instead of the source it means to
  test). `source` stays validate-on-read because a `MediaSource` string
  genuinely can be an out-of-domain value; a hash we wrote cannot. (Considered
  and rejected the validate-on-read alternative the review floated; trusted-wrap
  is both peer-consistent and lower-risk.)
- The SQL migrations (`0012_create_media.sql`, both backends) are **unchanged**
  ‚Äî the column stays `TEXT`; `ContentHash` is a Rust-side wrapper over the same
  stored string. No migration.

### `server/` ‚Äî compute, upload/serve DTOs, AtomPub

- `server/src/media_manager.rs`:
  - `UploadMetadata.sha256_hex: ContentHash`.
  - The two **compute sites** (`upload_bytes` and `stream_to_temp`) build the
    hash via `ContentHash::from_digest(format!("{digest:x}"))`; `stream_to_temp`
    returns `(ContentHash, u64)`.
  - `register_in_db(..., sha256_hex: &ContentHash, ...)`;
    `MediaRecord { sha256: ‚Ä¶ }` takes the `ContentHash` directly (`.clone()`
    where needed).
  - `media_path("upload", &metadata.sha256_hex, ‚Ä¶)` and the
    `UploadResponse { sha256: metadata.sha256_hex, ‚Ä¶ }` flow the typed value.
- `server/src/media.rs`:
  - `UploadResponse.sha256: ContentHash` (JSON `Serialize` DTO ‚Äî serializes as
    the same plain string via the trailer, so the JSON shape is unchanged).
  - `ServeParams.hash` **stays `String`** ‚Äî it is the raw axum `Path` extractor
    for an attacker-controlled URL segment with a **deliberate 404-on-invalid**
    contract (a DoS-safe "not found" that does not reveal validation). Typing
    the field would make axum's extractor reject a malformed hash with **400**
    before the handler, breaking that contract and the tests. Instead,
    `validate_serve_params` parses it into a `ContentHash` at the validation
    chokepoint and returns `(MediaSource, ContentHash)`; the
    `is_valid_content_hash` check is replaced by
    `params.hash.parse::<ContentHash>().map_err(|_| NOT_FOUND)` (same 404).
    `serve_response` uses the typed `ContentHash` for the ETag (`Display`),
    `find_by_hash`, and the path. This is ADR-0063 ¬ß4 "parse at the outermost
    **usable** boundary" ‚Äî the extractor is raw wire, the chokepoint is typed.
  - `resolve_media_path` (the sole caller of `validate_serve_params`) threads
    the parsed hash out: it returns `(MediaSource, ContentHash, PathBuf)` (was a
    2-tuple), so `serve_response` uses the typed `ContentHash` for the ETag,
    `find_by_hash`, and path (`params.hash: String` is used only to build it).
    **This breaks the inline test** in `server/src/media.rs`'s `mod tests`
    (`resolve_media_path_builds_path_for_valid_params`, ~:365):
    `let (source, path) = ‚Ä¶` ‚Üí `let (source, _hash, path) = ‚Ä¶`. (The `Err(...)`
    assertions at ~:386‚Äì439 still compile ‚Äî `Ok` is the only affected variant.)
  - `find_by_hash(&hash, &source)` receives `&ContentHash`.
- `server/src/atompub/media.rs`:
  - `media_link_entry` uses `record.sha256` (now `ContentHash`) in
    `media_url`/edit URI unchanged (via `Display`/`Deref`).
  - The POST-idempotency **compute site**
    (`let sha = format!("{:x}", Sha256::digest(&body))`) becomes
    `ContentHash::from_digest(format!("{:x}", ‚Ä¶))`; `get_media(‚Ä¶, &sha, ‚Ä¶)`.
  - `member_get`/`member_delete` take the hash from a URL `Path` segment
    (untrusted): parse it into `ContentHash` at handler entry, mapping a parse
    failure to the handler's existing not-found/invalid response (mirroring
    `ServeParams`; exact status confirmed against the handler during
    implementation). `upload.sha256` (from a parsed request body) flows typed.

### `web/` ‚Äî wire DTO, server fn, form

- `web/src/media/mod.rs`:
  - `MediaItem.sha256: ContentHash` (wire `Serialize`/`Deserialize` DTO ‚Äî
    serializes as the same plain string, wire shape unchanged). `list_my_media`
    maps `r.sha256` (now `ContentHash`) straight in;
    `media_url(‚Ä¶, &r.sha256, ‚Ä¶)`.
  - `#[server] delete_media(sha256: ContentHash, filename: String, source: String, force: Option<bool>)`
    ‚Äî **typed wire arg** per ADR-0065 (replacing the `String`+parse-on-entry
    stopgap that predates ADR-0065). Only `sha256` is typed (the #459 value);
    `source`/`filename` stay `String` (other scope). Note the current body does
    **not** parse `sha256` (it uses the raw `String` unvalidated in
    `media_url`/`delete_media`), so there is nothing to "remove" ‚Äî the typed arg
    _adds_ deserialize-time validation (defense-in-depth). The body just uses
    `&sha256` (now `&ContentHash`) at `media.delete_media(‚Ä¶, &sha256, ‚Ä¶)` and
    `media_url(‚Ä¶, &sha256, ‚Ä¶)`.
- `web/src/pages/media.rs`:
  - `render_media_row` reads `item.sha256` (now `ContentHash`); the hidden
    `<input name="sha256" value=‚Ä¶ />` renders the **owned** string form
    `value=item.sha256.to_string()` (`ContentHash` is not a leptos
    `IntoAttributeValue`; use `.to_string()` ‚Äî an owned `String` ‚Äî not
    `.as_ref()`, whose borrow would be captured into the `ActionForm` closure).
    The local `let sha256 = item.sha256.clone();` becomes
    `item.sha256.to_string()`.
  - **No `<ValidatedInput>` is introduced.** ADR-0065's client pre-validation
    exists to protect the _error UX of a user-typed field_. Here `sha256` is a
    **hidden, machine-populated** field whose value originates from an
    already-validated `MediaItem.sha256: ContentHash` returned by the server ‚Äî
    it can never be invalid on a legitimate client, so pre-validation is
    satisfied **structurally** and there is no field for a user to get wrong.
    The typed arg still `Deserialize`-validates server-side (ADR-0065
    defense-in-depth). This is the correct reading of ADR-0065 for a
    non-interactive field; the alternative (keep `sha256: String`, parse in the
    body) is exactly the obsolete stopgap the ADR retires.

### Test surface ‚Äî enumerated (the type change bites hardest here)

The type change forces edits in test files that are **not all mechanical**.
Every one is in scope (per the test-helper convention, media-hash fixtures use
`parse_content_hash`). Two classes of non-obvious break, called out explicitly:

- **`ContentHash` has `PartialEq<str>`/`<&str>` but NOT `PartialEq<String>`**
  (the `StrNewtype` trailer, `macros/src/str_newtype.rs:97-109`). So
  `assert_eq!(record.sha256, some_owned_string)` **will not compile**. The fix
  differs by site ‚Äî **do not blanket-apply `.as_ref()`**:
  - `server/tests/storage/mod.rs:6890,7184` ‚Äî the `sha256` local is **also
    passed to `get_media`/`find_by_hash` (`:6885,7180`)**, which now take
    `&ContentHash`, so `&String` no longer coerces: the local **must become a
    `ContentHash`** (`let sha256 = parse_content_hash("‚Ä¶");`). Once it is, the
    assertion is `assert_eq!(fetched.sha256, sha256)` ‚Äî
    **`ContentHash == ContentHash`, no `.as_ref()`** (adding `.as_ref()` here
    would be `&str` vs `ContentHash`, which has no `PartialEq` and fails to
    compile). This is the whole point: the type change makes the local a
    `ContentHash` and the comparison Just Works.
  - `server/src/media_manager.rs:590` ‚Äî here `expected_sha` is a genuine
    `format!` `String` (`:584`) with no `ContentHash` counterpart, and
    `first.sha256` is a `ContentHash`. Compare `&str` to `&str`:
    `assert_eq!(first.sha256.as_ref(), expected_sha.as_str())` (unambiguous).
    (Line 600 `second.sha256 == first.sha256` is fine ‚Äî std `PartialEq<Self>`.)
- **Fixtures with sub-64-char hashes must become real 64-char hashes** wherever
  the value now round-trips through `ContentHash` decode (a typed wire arg, a
  `find_by_hash`/`get_media` call, or `media_url`), because a short string fails
  `FromStr`. Files: `server/tests/storage/mod.rs` (`make_media_record` at :6845
  takes `&str` ‚Üí `&ContentHash` or builds via `parse_content_hash`; call sites
  :6885,6944,6975,7180; literals throughout), `server/tests/web/web_media.rs`
  (:61,124,175,229,249,284,287,324 ‚Äî `MediaRecord` fixtures, `media_url`
  literal, and `delete_media` form bodies),
  `server/tests/misc/backup_fixture.rs:16-17,165` (`FIXTURE_MEDIA_SHA256` is
  already 64-char ‚Äî verify), `server/tests/misc/ media_handlers.rs`
  (`SAMPLE_HASH` already 64-char ‚Äî verify), and the `cfg(test)` blocks in
  `storage/src/media.rs:399,418,439,460` (`"abc123"` ‚Üí 64-char), and the
  **`server/src/media.rs` inline `mod tests`**
  (`resolve_media_path_builds_path_for_valid_params` ~:365 ‚Äî the 3-tuple
  destructure above; the `Err` cases are unaffected). The
  `storage/src/helpers.rs:728,744` row tests keep their short `"sha256"`
  placeholder **because** the read path trusted-wraps (above), so they do
  **not** break ‚Äî a deliberate benefit of the trusted-wrap choice.
  (`FIXTURE_MEDIA_SHA256` and `media_handlers.rs`'s `SAMPLE_HASH` are already
  64-char.)

## Behavior changes (accepted, defense-in-depth only)

Per ADR-0065, typing a wire arg moves its validation into arg-**decode**:

1. `web::delete_media` ‚Äî a malformed `sha256` (only reachable by a non-browser
   client; the browser only ever submits a valid `MediaItem.sha256`) now fails
   at server-fn decode as a generic transport error **before** the `boundary!`
   body, instead of the body's `MediaSource`/parse error path.
   - **Do not blindly "assert non-OK" on the affected tests** ‚Äî one of them
     tests a _different_ thing. `server/tests/web/web_media.rs:61`
     (`media_endpoint_rejects_unauthenticated_request`) posts `sha256=deadbeef`
     (invalid) and asserts `500` + body `"unauthorized"` ‚Äî it proves the **auth
     gate** rejects. Under a typed arg, decode fails before `require_auth`, so
     the body no longer says "unauthorized" and the test's _purpose_ is
     destroyed. Fix the opposite way: give this case a **valid 64-char hash** so
     it still exercises the auth path. Only a test whose _intent is the hash
     rejection_ switches to "assert non-OK".
   - The `delete_media` success / reference-check tests (`web_media.rs:249,324`)
     use 16-char hashes in both the `MediaRecord` and the form body and assert
     `OK`/`deleted`; they need valid 64-char hashes **throughout** (fixture and
     form body must match) or they fail decode.
2. `ServeParams` / AtomPub member routes ‚Äî **no** change: the hash is parsed at
   the chokepoint (not the extractor), so a malformed hash still yields the
   existing 404/not-found, not a 400.

No other observable behavior changes. Wire/JSON/SQL shapes are byte-identical.

## Out of scope (explicitly not touched)

- `user_id` (‚Üí `UserId`, #471), `filename`, `MediaSource`-as-wire-`String`
  (`delete_media`'s `source` arg, `MediaDialect::delete_media_row`'s
  `source: &str`).
- The **NOT-IN-SCOPE** non-media hashes: feed/post/page ETags (`"sha256-‚Ä¶"`
  validators in `common/src/feed/metadata.rs`, `server/src/atompub/posts.rs`,
  `server/src/projector/mod.rs`), backup file integrity hashes
  (`storage/src/backup.rs` + backend `backup.rs`), and the session `TokenHash`
  (`host/src/token.rs`). These are different concepts and are left untouched.
  (The media-serve ETag, which _is_ derived from the content hash, is in scope
  and flows the `ContentHash` via `Display`.)
- No new ADR (this applies the existing ADR-0063 convention, as `Email`/#458
  did); no schema migration; no `xtask` gate (not a security newtype).
